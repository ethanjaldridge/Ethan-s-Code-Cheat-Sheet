Tags
Tag Name Description
<head> Head Contains the metadata for a webpage, such as links to .css files and the webpage title.
<title> Title A browser’s tab displays the title specified in the title tags. The title tag is always inside of the head.
<header> Header A container for introductory content, such as h1-h6 and nav links.
<nav> Navigation Defines a block of navigation links (menu, table of contents, services, contact, etc.) Can be used inside of the header element, or on its own. Inside the <nav> tags, create an unordered list. Inside the <ul> tags, create list items <li>. Inside the <li> tags, create anchors for links <a href="#nameofsectionid">. Name the link inside of the <a> tags, just before the closing tag.
<main> Main Contains all of the content of a webpage (similar to body). Separate from nav and footer. May contain header.
<body> Body Contains all of the content that is displayed on the webpage.
<h1> Heading Creates a heading for a section on the webpage. Values range from h1-h6, which are used to determine the size of the text. h1 is the largest; h6 is the smallest.
<p> Paragraph Creates a paragraph; a block of plain text.
<article> Article Holds content that makes sense on its own: articles, blogs, comments, etc.
<section> Section Defines elements in a document such as chapters, headings, etc. Sections usually contain articles, but this can go either way.
<aside> Aside Marks additional information that enhances another element but isn’t required to understand the main content. Can be used alongside other elements such as section and article. Used for: bibliographies, endnotes, comments, pull quotes, editorial sidebars, and additional information.
<span> Span Encloses short pieces of text as a means to separate content on the same line.
<em> Emphasize Emphasizes text in italics.
<strong> Strong Emphasizes text in bold.
<br> Line Break Also referred to as newline, line break creates a new line of text...
and does not require a closing tag.
<ul> Unordered List Creates an unordered (bulleted) list. Individual list items must be enclosed in <li> tags, otherwise they will not appear in the list on the webpage.
<li> List Item Used to describe an item in a list.
<ol> Ordered List Creates an ordered (numbered) list.
<a> Anchor Emphasizes text that serves as an "achor" for a link. Links can take you to the same page, an internal page, or an external page. Common attributes are href and target.
<button> Button Creates a button that a user can click. Buttons commonly appear as rectangles, rounded rectangles, or circles.
<div> Division Creates a container that separates content from other content on the same page.
<figure> Figure Encapsulates images, figures, illustrations, diagrams, etc. Can be moved anywhere in a document.
<figcaption> Figure Caption Describes the media in the figure. This code is usually written inside of <figure>.
<img> Image Inserts an image onto the webpage. Self-closing tag; can include or exclude the final / before the >. Requires src attribute. Wrapping images with an anchor attribute turns the image into a link.
<video> Video Inserts a video onto the webpage. Requires closing tag and src attribute. Useful attributes include controls, autoplay, and loop. "Video not supported" will only display if the browser is unable to support the video.
<audio> Audio Encapsulates audio links. Can be written as <audio src="audiolink"> or <source src="audiolink"> within the audio tags. Helps to include the type attribute.
<table> Table Inserts a table.
<tr> Table Row Inserts a table row.
<td> Table Data Inserts table data. Each new data point in a row creates a new column in that row.
<th> Table Header Defines a title for columns and rows. The scope attribute is used to indicate a column <col>, or row <row>.
<thead> Table Head Allows the table's column titles to be sectioned off.
<tbody> Table Body Allows the table's body to be sectioned off.
<tfoot> Table Foot Allows the table's footer to be sectioned off. Table footers are often used to display sums, differences, etc.
<footer> Footer Contains information such as: contact, copyright, terms of use, site map, reference to top of page links, etc.
<style> Style Allows you to write CSS code within the <head> of an HTML file.
<!----> Comment Inserts a comment into your code. Any characters in-between the -- -- are ignored by your browser and will not appear on the webpage. Shortcut: Ctrl+/
<> Less than, Greater than To display the less than (<) and greater than (>) symbols when discussing code tags on a webpage, without creating a code prompt (see far left), use & l t ; for (<) and & g t ; for (>).
<sup> Superscript Makes the text superscripted.
& thinsp ; Thin Space Creates a narrow space between the text (type as one word).
& nbsp ; Space Creates a regular space between the text (type as one word).
& ensp : Double Space Creates two spaces between the text (type as one word).
& emsp ; Quadruple Space Creates four spaces between the text (type as one word).
<blockquote> Blockquote Around a paragraph element that contains a quote, this italicizes the text and creates a (usually) grey, vertical bar to the left of the quote.
<hr> Horizontal Rule The <hr> HTML element represents a thematic break between paragraph-level elements: for example, a change of scene in a story, or a shift of topic within a section.

Links
Type Description
Same Page Create an id for a section of the webpage. Then, create a link to the id that looks something like this: <a href="#nameofid">Nameofid</a>. In the #nameofid, the name is case sensitive and must match the capitalization in the id itself.
Relative Path A relative path is a filename that shows the path to a local file. A local file is a file in the same folder. A relative path links to another page on the same website. An example would be <a href="./contact.html">Contact</a>, where the "./" tells the browser to look for the file in the same folder.
Absolute Path An absolute path links to a file that is stored in a different folder. An absolute path links to a webpage on a different website, and requires a full URL in the href link.
Web Fonts You can link to outsourced fonts from your HTML document with a <link>. Go to a font hosting website (Google Fonts, Adobe Fonts, fonts.com). Pick a font. Pick a style. The <link> is automatically generated for you to use. Add the <link> to the <head> of your html. Create font-family declarations in your CSS using quotes (" ") around the name. Ex: "Macondo".
Web Fonts (@font-face) You can create a @font-face ruleset in your CSS stylesheet to link to the relative path of the font file. Font file formats (in order of least to most progressive): OTF (OpenType Font), TTF (TrueType Font), WOFF (Web Open Font Format), WOFF2 (Web Open Font Format 2). It's a good idea to include TTF, WOFF, and WOFF2 formats with your @font-face rule to ensure compatibility on all browsers. Go to a font hosting website. Select a font + select a style (opens, “Selected Families”). Within Selected Families, click Download. Files will be downloaded in a single format. Move the font files into a folder in your project's working directory. You are now ready to use the font files in a @font-face {font-family: ;} ruleset. Define the font-face ruleset at the top of your CSS stylesheet. font-family is used to set a custom name for the font, in quotes "". In CSS, use font-family: "nameofyourfont".

Form Validation Validation is the concept of checking user provided data against the required data. Most data, once submitted, is stored by a website or web application (data submitted through forms are stored as strings). It’s stored in a database on the server side. There are important reasons for us to make sure the information that will be stored in the database is accurate (to prevent username doubling, address error in shipping, leaking of personal info, etc.)
Regular Expressions To specify patterns for the computer to recognize, we use a special language called regular expressions — also known as regex or regexp. A regular expression is a sequence of characters representing a pattern. We can use that pattern to match a string, match parts of a string, confirm that data is formatted acceptably, or even replace parts of strings with different characters. More on regex
Client-side Validation: HTML Client-side validation occurs when problematic inputs are prevented from being submitted in the first place. The client is the process interacting with the server on behalf of a user—in the case of websites, the web browser is the client. The logic for validating the form is included with the code that displays the form on the user’s device. No interaction with the back-end is required to perform client-side validations. HTML provides some of these validation features built-in (required fields, min and max values, input must match a particular pattern specified by a regular expression).
<input id="allergies" name="allergies" type="text" required> required Adding the required attribute to an <input> element makes it so that the field must be completed in order to submit the form. An error message will appear if the user leaves the field blank.
<input id="guests" name="guests" type="number" min="1" max="4"> min max Another built-in validation we can use is to assign a minimum or maximum value for a number field (type="number" and type="range").
<input id="summary" name="summary" type="text" minlength="5" maxlength="250" required> minlength maxlength We can also set a minimum and/or maximum character count for a text field.
<input id="payment" name="payment" type="text" required pattern="[0-9]{14,16}">

pattern="[cC]odecademy" 		pattern For cases when we want user input to follow specific guidelines, we use the pattern attribute and assign it a regular expression, or regex. Let’s say we wanted to check for a valid credit card number (a 14 to 16 digit number). We could use the regex: [0-9]{14,16} which checks that the user provided only numbers and that they entered at least 14 digits and at most 16 digits.
Client-side Validation: JavaScript In order to truly customize validation or to perform more complex validations, we can incorporate JavaScript form validations. We can do this by either writing the JavaScript ourselves or by incorporating a JavaScript library. Most basic validation libraries will involve directly accessing or manipulating the DOM. This can get tricky when working with a framework that relies on a virtual DOM, like React or Vue. In those situations, it might be best to incorporate a library that works well with your specific framework.
Server-side Validation Validations must also be completed on the back-end or server-side. As a rule, the back-end should never trust the data it receives. Front-end validations are easy to bypass. A malicious user can simply turn off JavaScript on their browser, for example. There’s also the potential for middleman attacks in which data is changed after the request is submitted by a user but before it arrives at the server.

Advantages of server-side validation:

  • Once data is in the back-end, the developer has complete control over it.
  • Developers can use validation code, often on machines with more computing power.
  • Developers can write validation code that a user can’t see (more difficult for malicious users to hack).
  • Developers can validate the information against other data the front-end doesn’t have access to (is the username taken?).


There are two main ways to validate inputs on the server-side. The first takes place while the user is still inputting data into the form on the front-end. We can make asynchronous requests to the server with pieces of their data and send feedback directly to the user before they’ve submitted. This is slower than front-end validation and can be a design challenge from a user-experience perspective.

The second is once the form has been submitted. Back-end form validation is our application’s last defense against problematic data, and it’s essential to verify the validity and safety of data before adding it to a database. This is also an opportunity to “sanitize” the data: in order for our database to be useful, it’s important that all data within it is formatted consistently. This means that while we may want to be flexible about the formatting we require from a user, we likely want to transform inputs into a strict format before entering them in the database.
